# Buoy 0.4 Console Alias Removal

## Purpose and scope

Remove the deprecated installed `turbo-search` console command in Buoy 0.4.0 while preserving the primary `buoy` command and every separately governed compatibility surface.

## Behavior

- Distribution metadata for 0.4.0 MUST expose `buoy = buoy_search.cli:main` and MUST NOT expose a `turbo-search` console script.
- The dedicated `buoy_search.cli:legacy_main` compatibility hook MUST be deleted, including imports and tests whose only purpose is that hook. The primary `main` parser and dispatch path MUST remain the console implementation.
- A clean 0.4.0 wheel installation MUST contain a working `buoy` launcher and MUST NOT contain a package-owned `turbo-search` launcher.
- Upgrading one isolated environment from the released GitHub 0.3.0 wheel to the candidate 0.4.0 wheel MUST keep `buoy` functional and MUST remove the package-owned `turbo-search` launcher generated by the 0.3.0 installation.
- Scripts migrate by replacing only executable `turbo-search` with `buoy`; arguments, parser behavior, output, exit codes, direct-command defaults, and safety gates MUST NOT change because of this removal.
- The package makes no claim about user-created shell aliases, copied launchers, wrappers, caches, or files outside package-manager ownership.

## Package and wheel validation

Validation MUST use temporary directories and an isolated environment and MUST NOT install either artifact into the developer environment.

1. Build the candidate wheel from the exact implementation commit and assert its distribution version is `0.4.0`.
2. Inspect the candidate wheel's `entry_points.txt`: it MUST contain exactly the primary `buoy = buoy_search.cli:main` mapping for this product and no `turbo-search` mapping; the wheel MUST contain required `buoy_search` code/data and no dedicated legacy hook.
3. For clean-install coverage, install the candidate wheel in a fresh isolated environment, run `buoy --version` and `buoy --help`, and prove no package-owned `turbo-search` launcher exists.
4. For upgrade coverage, download the immutable released artifact `https://github.com/Doctacon/buoy-search/releases/download/v0.3.0/buoy_search-0.3.0-py3-none-any.whl`, verify SHA-256 `048dba11df692a7efcd7ab7269fc2eec82f6b53e57573a3de113bbd051750bab`, and install that wheel into a second fresh isolated environment.
5. In that same second environment, prove distribution version 0.3.0, `buoy --help`, and the package-owned `turbo-search --help` launcher are present; then run the package manager's normal upgrade operation against the candidate 0.4.0 wheel without recreating the environment.
6. After the upgrade, prove distribution version 0.4.0, `buoy --version`, and `buoy --help`; inspect installed distribution entry points and the environment's launcher directory to prove the package-owned `turbo-search` launcher is absent. Merely testing a clean 0.4.0 install is insufficient upgrade evidence.

Dependency downloads, if needed for isolated installation, are reads only. Validation MUST NOT publish a package, create a tag/release, or invoke Turbopuffer.

## Retained compatibility

This removal MUST NOT alter:

- `.turbo-search` state-root fallback or explicit `--state-root` selection;
- schema-supported old plans or pre-rebrand paths and identifiers;
- retrieve `--auto-route`, `--live`, or preview alias `--plan`;
- plan positional/`--base-url` source compatibility;
- apply `--plan PATH`;
- evals `--live`;
- `buoy catalog migrate-local`;
- deterministic row IDs, plan/apply IDs, remote namespaces, rows, cards, or artifact hashes;
- `BUOY_*` or `TURBOPUFFER_*` configuration.

Environment-alias removal is governed separately by `.10x/specs/buoy-v0-4-environment-alias-removal.md`.

## State, data, and external effects

Removing the console alias MUST perform no state discovery, copy, move, merge, rewrite, migration, re-embedding, local deletion, remote read/write, namespace mutation, tag creation, package publication, or release creation. Build/install validation is limited to disposable temporary locations.

## Acceptance scenarios

- Given a clean candidate 0.4.0 wheel install, when console entry points and launchers are inspected, then `buoy` exists and `turbo-search` does not.
- Given the released, digest-verified 0.3.0 GitHub wheel in an isolated environment, when that same environment is upgraded normally to the candidate 0.4.0 wheel, then the old package-owned launcher is absent and `buoy --help` works.
- Given `buoy --help`, `buoy --version`, `python -m buoy_search --help`, or the existing command argument matrix, when the console alias removal lands, then primary parser/help/version/dispatch behavior is unchanged.
- Given existing `.turbo-search` state or an old schema-supported plan, when it is used through `buoy`, then its separately governed compatibility remains intact and no data is moved or rewritten.
- Given built wheel metadata and source inspection, when checked, then neither a `turbo-search` entry point nor `legacy_main` remains while required `buoy_search` package data remains present.

## Acceptance criteria

- `pyproject.toml` has no `turbo-search` script entry and `src/buoy_search/cli.py` has no dedicated `legacy_main` hook.
- Focused source, parser, package-metadata, clean-install, and same-environment 0.3.0-to-0.4.0 upgrade tests pass.
- The released 0.3.0 wheel URL and digest are verified before the upgrade test.
- Candidate and installed metadata report 0.4.0; installed entry points and launcher-directory inspection prove absence rather than inferring it from source.
- Primary command behavior and every retained compatibility surface remain unchanged.
- Documentation and changelog migration guidance say to replace only `turbo-search` with `buoy` and do not imply deletion of user-owned launchers.
- Evidence records exact commands, artifact hashes, isolated paths, entry-point/launcher observations, focused/full test results, distribution checks, and an explicit no-state/no-remote/no-publication attestation.

## Explicit exclusions

Environment-alias implementation; removal of any retained compatibility; arbitrary user-owned launcher deletion; state/data migration; unrelated parser refactoring; PyPI publication; Git tag or GitHub Release creation; live Turbopuffer operations.
